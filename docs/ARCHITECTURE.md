# 架构设计（ARCHITECTURE）

> 本次对话敲定的长远平台架构。用「线上办公楼」比喻。

## 终极形态：线上办公楼（5 个楼层）

| 楼层 | 职责 | 关键点 |
|---|---|---|
| **Playground** | 分析主操作台 | 选不同**分析管线**（按行业 Know-how 定制，处理/评分/评价各异）；管线**热插拔**、可接市场、可收录用户自制；含 **Agentic Mode**（中心 ReAct 智能体自解析项目→判行业→定计划→调子 Agent→设计评分，全自主动态适配） |
| **Flip** | 批量过项目（投资人等） | 丢入一堆项目→**后台异步批量**跑（领域自选）→批量产出"**体检单**"审查报告→支持问答 |
| **Project Base** | 项目迭代工作区 | 上传计划书/资料一站式迭代创新项目，平台给辅助工具。**类比 Claude Code 工作区** |
| **Studio** | 创作台 | 用户设计自己的管线、报告形式 |
| **Marketplace** | 插件市场 | 收录用户设计品，应用到自己的 Novoscan |

> 旧库的 **Bizscan/Matchscan/Clawscan 未来不再是独立产品，而是 Playground 里的预置管线**。

## 架构剖面：一个心脏 + 两根柱子 + 记忆层 + 地基

```
🏛 外壳：Claude Desktop 式分板块导航（大堂与电梯）
🏢 楼层：Playground · Flip · Project Base · Studio · Marketplace
❤️ 心脏：管线引擎  —— 全楼共享的"分析大脑"
     管线定义 · 编排调度 · 热插拔 · Agentic(ReAct)
     + 共享能力：双轨检索 · AI客户端 · 评分体系 · 报告渲染
🟦 柱①：项目/工作区状态层   → 给 Project Base（托管/文档/迭代历史/协作）
🟧 柱②：创作与分发体系       → 给 Studio + Marketplace（作者工具/插件契约/沙箱/上架）
🧠 记忆与知识沉淀层（横跨心脏与地基）
     每次分析/体检单沉淀 · 行业知识库 · 用户专属记忆（越用越懂你）→ RAG 喂给 Agent
🧱 地基：账号 · 计费 · 数据库(含向量) · 异步任务队列 · 文件存储
```

**怎么读这张图**：
- **心脏（管线引擎）**是 Playground/Flip 共用的同一个分析大脑——前者交互式跑、后者后台批量跑。
- **柱①** 是 Project Base 的"例外"：它要长期托管项目、记住迭代历史，故在引擎之外额外需要一层状态。
- **柱②** 是 Studio（造管线）+ Marketplace（卖/装管线）的创作生态，围着引擎转但本身独立。
- **记忆层** 让系统越用越懂行业/用户，是专业用户复购的核心。

## 施工顺序（已与用户确认）

1. **引擎 + Playground 首条管线**（免登录可试）← 先盖第一层
2. **Flip** + 所需地基（账号 + 任务队列 + 基础计费）+ 批量闭环 ← 提前（投资人批量=早期付费点）
3. **Agentic Mode**
4. **更多预置管线**（Bizscan 等迁为管线）
5. **Studio + Marketplace**（创作生态，最重、最后）
- **Project Base**：灵活插入（与 2–4 并行或更后，看资源）

## 4 条设计铁律（第一行代码就要守，否则将来楼层接不上）

1. **管线是一等公民**：可声明、可版本化、可热插拔（Studio/Marketplace 才接得上）。
2. **引擎与界面分离**：引擎是纯净 TS 包，能被无界面/批量调用（Flip 才跑得起来；也是"运行基座可平移"的前提）。
3. **分析核心无状态，状态单独一层**（Project Base 才包得住）。
4. **插件契约与沙箱早立规矩**（哪怕第一版不开放）。

## 双轨执行与工具层(2026-06-10 增补)

平台分析能力支持两种执行方式,**共享同一套"工具"**:
- **固定管线(可信模式)**:按声明式配方(`PipelineDefinition`)跑,流程透明、可复现、成本可控 —— 对应"可信度优先"定位。默认管线走这条。
- **Agentic 模式(灵活模式,施工第3步)**:把工具清单交给中心 ReAct 智能体,自主决定调哪些、什么顺序 —— 适合复杂/未预设场景。代价:可追溯性下降、更慢更贵。

**工具层统一(`EngineTool`)**:数据源、子 Agent 都封装成统一工具(名字 + 用途说明 + 入参 schema + 执行函数)。
- 固定管线按 id 引用工具 + 喂入参;Agentic 模式经 `core/tools/ai-sdk-adapter` 转成 AI SDK tool 交给主控 AI。
- `execute` 同一个,两种模式**零重复、零返工**。
- 呼应铁律①(管线一等公民)与④(插件契约早立):工具即插件的统一形态。
- 落地:`core/tools/`(契约 + 注册表 + 适配器);阶段 4 数据源、阶段 5 子 Agent 均实现 `EngineTool`。

## 证据闸门:治"垃圾进垃圾出"(2026-06-10 增补)

旧库检索"搜到啥用啥",召回后无相关性过滤 → 噪声证据喂给 Agent → 结论不扎实。新架构在 **检索完成 → 交给 Agent 之前** 强制经过一道证据闸门(`core/search/rerank.ts`):
1. **相关性重排 + 过滤**:每条候选按"与用户创意的真实相关性"打分,砍低相关、按分排序。
2. **质量加权**:相关性 × 权威度(学术看引用/期刊,产业看 star/来源)。
3. **充分性闸门**:高相关证据过少时标记"证据不足/低置信",透明披露而非硬编结论。

打分策略可插拔:**LLM 重排**(国产模型,跨语言语义判断,主力) → **规则重排**(关键词命中+引用+时近,无 key 兜底、仅排序不过滤) → **向量重排**(embedding 相似度,阶段6)。这是"可信度优先"在数据源头的保障。
